Overview

Check Lib has two working areas.

Runner App

Used to complete inspections, capture issues, sign off work, and export reports.

Checklist Builder

Used to create, edit, import, duplicate, and delete custom checklists stored on the device.

The app is local-first. Built-in checklists load from the /checks folder. Custom checklists are stored in IndexedDB and then merged into the Runner checklist selector.

Start a checklist

  1. Open the Runner app.
  2. Select the checklist from the drop-down list.
  3. Enter the date.
  4. Enter the Team Leader or Manager name.
  5. Enter the Team Member leading the task.
  6. Enter the Unit or Area.
  7. Tap Start Checklist.

The stage bar moves from Setup to Sections, then Issues, then Final.

Complete sections

Each section shows the current section title, item count, answered count, and issue count.

  • Use Sections to jump through the drawer navigator.
  • Use Show unanswered to focus only on incomplete items.
  • Use the sticky footer for Back, Save, Issues, and Next on mobile.
  • Autosave runs while you work after the checklist starts.

Review issues

An issue is created when an item is marked as a failure state. The Issues Review stage lets you complete the issue record before sign-off.

  • Add issue comments.
  • Set severity.
  • Mark whether action is required.
  • Assign a responsible person if needed.
  • Set a due date if needed.

Use the jump buttons to return to the related section item.

Final review and signature

The final stage checks readiness before signature. You cannot sign off until required fields and issue comments are complete.

  • Use Save Draft to hold progress.
  • Use Proceed To Signature when validation is complete.
  • Sign inside the signature pad and confirm.

Queue, export, and archive

Queue

Active reports waiting for export. You can export, archive, search, filter, inspect, and clear items.

Archive

Stores completed report history. You can re-export, restore, search, filter, and delete archived items.

Export modes:

  • Full report
  • Summary and issues only
  • Issues only

After export, choose whether to keep reports in queue or move them to archive.

See PDF export and file handling below for the full export flow, content modes, single report export, and CSV summary output.

PDF export and file handling

Check Lib uses the queue as the holding area for completed reports. A report is not part of the export batch until it has been added to the queue.

Add to Queue

Saves the current signed checklist as a queued report ready for export. This does not export the PDF yet.

Export Queue

Exports the reports currently stored in the queue. This is the main PDF output action for batched report handling.

Export content modes

  • Full report, includes summary, issues, and full checklist content.
  • Summary and issues only, keeps the result focused on overview and exceptions.
  • Issues only, outputs only the exception content for follow-up use.

After export behaviour

  • Keep in queue, the report stays in the active queue after PDF export.
  • Move to archive, the report leaves the queue and is stored in archive history after export.

Single report export

Queue cards and archive cards support single report export. Use this when you do not want to export the full queue.

Archive handling

  • Archived reports can be viewed again.
  • Archived reports can be re-exported.
  • Archived reports can be restored to the queue.
  • Archived reports can be deleted when no longer needed.

CSV export

Queue CSV export and Archive CSV export create summary files for report tracking. These are not the same as the PDF report output and are intended for list-style review and analysis.

PDF content notes

  • The PDF uses checklist metadata such as date, supervisor, team member, and unit or area.
  • Issues are pulled from failed items and include comments and issue fields entered during review.
  • Signed reports include the captured signature in the final exported record.

Builder guide

  1. Open Builder from the app header or direct link.
  2. Create a new checklist or select one from the library.
  3. Set title and description.
  4. Add sections.
  5. Add tasks inside each section.
  6. Save the checklist to local storage.

Import CSV

Imports a CSV file into the editor. Review the structure before saving.

Custom checklist library

Stored in IndexedDB on the device. Runner reads these and adds them to the checklist selector.

Offline, caching, and storage

  • The service worker caches core app files for repeat use.
  • Queue and archive data are held in localStorage.
  • Builder checklists are held in IndexedDB.
  • Built-in checklist files remain in the app package under /checks.

If the service worker cache fails during install, the app may still load from network but offline behaviour may be limited until the cache issue is fixed.

Troubleshooting

Custom checklist not showing in Runner

Save it in Builder, then refresh the Runner app so metadata reloads.

Save failed in Builder

Refresh the page and try again. Database connection closing errors usually happen during unstable IndexedDB state.

Offline install issue

Check the service worker file list. Missing or invalid cache paths will break cache.addAll.

Missing report

Check whether it was archived, deleted, or still held in queue.
